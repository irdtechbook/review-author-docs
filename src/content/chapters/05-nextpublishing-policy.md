ここでは「Re:VIEW として書けるか」と「Nextpublishing で通りやすいか」を分けて考えます。初心者が詰まりやすいのは、たいてい後者です。

## 禁止・非推奨タグ

### まず「代替案」を覚える

はじめて書籍化する著者は、理由を細かく追う前に、**何がだめで、何に置き換えればよいか** を先に知るほうが作業しやすくなります。

| 問題になりやすいもの | 先に覚える代替 | 覚え方 |
| --- | --- | --- |
| `//list` | `//listnum` | コード例はまず `listnum` から考える |
| 複雑な見出し装飾 | 飾らない見出し | 強調は本文へ逃がす |
| 深すぎる箇条書き | 見出し分割 | ネストで解決しない |

### 先に避ける NG の全体像

<div class="mermaid">
flowchart TD
  A[原稿を書く] --> B{詰まりやすい場所はどこか}
  B --> C[見出し]
  B --> D[コード例]
  B --> E[箇条書き]
  B --> F[表や画像]
  C --> G[飾りすぎを避ける]
  D --> H[list と listnum を混同しない]
  E --> I[深いネストを避ける]
  F --> J[ID と拡張子を確認する]
</div>

### `//list` を避けたい理由

Nextpublishing では、コード例まわりで修正が必要になりやすく、最初に迷いやすいのが `//list` です。最初から行番号付きの `//listnum` を使うと、説明しやすさと安定性を両立しやすくなります。

```text
//listnum[hello][hello.rb]{
puts "hello"
//}
```

### 見出しに修飾を入れない

見出しでは、強調、脚注、複雑な参照などの修飾を入れないでください。見出しに修飾を入れる書き方は、修正対象になる NG 記法です。

```text
== よい例

== API の基本
```

```text
== 避けたい例

== @<b>{API} の基本
```

### 非推奨の警告は「いま直せるなら直す」

警告だから後回しでもよい、とは限りません。運用では、警告の積み重ねが読みにくさや後の修正負担につながります。はじめて書籍化する著者ほど、警告を早めに潰すほうが後で楽になります。

## 見出し・参照・ID の制約

### 見出しは飾らない形で保つ

見出しは構造の要です。ここに装飾や参照を入れると、見た目のために構造が不安定になります。**見出しは飾らず、説明は本文で行う** 形にそろえてください。

### ID は「重複しない短い名前」

ID は一度付けると長く使います。そこで大切なのは、凝った命名よりも**重複しない・短い・役割が分かる** ことです。

### 未定義参照は「参照先」から直す

参照エラーが出たとき、参照を書く側ばかり見がちです。しかし実際には、参照先の ID が消えていたり、空だったりすることもあります。初心者は、**参照元と参照先の両方を見る** 習慣を付けてください。

### ID で起きやすい NG

ID でよく起きるのは、次の 3 つです。

| NG | 何が起きるか | 直し方 |
| --- | --- | --- |
| 同じ ID を何度も使う | どこを指すか曖昧になる | ID を章ごとに分ける |
| 長すぎる ID を付ける | 打ち間違いしやすい | 短く分かる名前にする |
| 参照先だけ消す | 未定義参照になる | 参照元も一緒に確認する |

## 箇条書き・表・画像の修正ポイント

### 箇条書きは見た目で判断しない

見た目で揃って見えても、空白や空行の違いでルールに引っかかることがあります。箇条書きが怪しいときは、本文の意味ではなく**記号の置き方** を見直してください。

### 表はセル区切りを疑う

表の警告は、内容が悪いより「区切りの解釈」が違うケースが多いです。詰まったらまずセル区切りの書き方を疑います。

### 画像は拡張子も確認する

本文参照が合っていても、拡張子要件で止まることがあります。画像系の診断は、本文だけ見ていても解決しないと覚えておくと切り分けが速くなります。

### 表と画像で起きやすい NG

| 対象 | よくある NG | 先に見る場所 |
| --- | --- | --- |
| 表 | セル区切りの崩れ | 表の区切り記号 |
| 表 | 説明文がない | 表の定義部 |
| 画像 | 参照 ID と実ファイル名がずれる | 本文と `images/` |
| 画像 | 拡張子要件を外す | 画像ファイル名 |

## 環境依存と描画系依存の警告

### 環境依存文字は「通る端末」と「通る組版」が違う

自分のエディタでは見えていても、組版では扱いづらい文字があります。はじめて書籍化する著者は、「見えたから安全」と考えず、警告が出たら文字そのものを見直してください。

### Mermaid の警告は図の複雑さを下げる合図

Mermaid は便利ですが、複雑になるほど差異が出やすくなります。警告が出たら、文法だけでなく**図の役割が大きすぎないか** も見直してください。

### Mermaid で避けたい書き方

はじめて書籍化する著者は、1 枚の図に全部を詰め込まないほうが安全です。目安は次のとおりです。

- ノード数を増やしすぎない
- 矢印の交差を増やしすぎない
- 章の流れ 1 つだけを描く

## 直し方の優先順位

1. 代替案が明確なものから直す
2. 見出しや ID のような構造要素を先に直す
3. 画像や表など周辺ファイルを伴うものを確認する
4. 警告が消えたあとに本文の読みやすさを整える

この順にすると、構造の根本原因を先に潰しやすくなります。
