ここでは「Re:VIEW として書けるか」と「Nextpublishing で通りやすいか」を分けて考えます。初心者が詰まりやすいのは、たいてい後者です。

## 禁止・非推奨タグ

### まず「代替案」を覚える

禁止や非推奨の話は、理由から入ると重たくなります。初心者向けには、先に**何へ置き換えればよいか** を示したほうが役立ちます。

| 問題になりやすいもの | 先に覚える代替 | 覚え方 |
| --- | --- | --- |
| `//list` | `//listnum` | コード例はまず `listnum` から考える |
| 複雑な見出し装飾 | plain text の見出し | 強調は本文へ逃がす |
| 深すぎる箇条書き | 見出し分割 | ネストで解決しない |

### 非推奨 warning は「いま直せるなら直す」

warning だから後回しでもよい、とは限りません。運用では、warning の積み重ねが読みにくさや差し戻し予備軍になります。初心者ほど、warning を早めに潰すほうが後で楽になります。

## 見出し・参照・ID の制約

### 見出しは plain で保つ

見出しは構造の要です。ここに複雑な装飾や参照を入れると、見た目のために構造が不安定になります。初心者向けには、**見出しは plain、説明は本文** という役割分担を勧めます。

### ID は「重複しない短い名前」

ID は一度付けると長く使います。そこで大切なのは、凝った命名よりも**重複しない・短い・役割が分かる** ことです。

### 未定義参照は「参照先」から直す

参照エラーが出たとき、参照を書く側ばかり見がちです。しかし実際には、参照先の ID が消えていたり、空だったりすることもあります。初心者は、**参照元と参照先の両方を見る** 習慣を付けてください。

## 箇条書き・表・画像の差し戻しポイント

### 箇条書きは見た目で判断しない

見た目で揃って見えても、空白や空行の違いで rule に引っかかることがあります。箇条書きが怪しいときは、本文の意味ではなく**記号の置き方** を見直してください。

### 表はタブ区切りを疑う

表の warning は、内容が悪いより「区切りの解釈」が違うケースが多いです。詰まったらまずセル区切りの書き方を疑います。

### 画像は拡張子も確認する

本文参照が合っていても、拡張子要件で止まることがあります。画像系 diagnostics は、本文だけ見ていても解決しないと覚えておくと切り分けが速くなります。

## 環境依存と renderer 依存の警告

### 環境依存文字は「通る端末」と「通る組版」が違う

自分のエディタでは見えていても、組版では扱いづらい文字があります。初心者向けには、「見えたから安全」と考えず、warning が出たら文字そのものを見直す姿勢を勧めます。

### Mermaid warning は図の複雑さを下げる合図

Mermaid は便利ですが、複雑になるほど差異が出やすくなります。warning が出たら、文法だけでなく**図の責務が大きすぎないか** も見直してください。

## 直し方の優先順位

1. 代替案が明確なものから直す
2. 見出しや ID のような構造要素を先に直す
3. 画像や表など周辺ファイルを伴うものを確認する
4. warning が消えたあとに本文の読みやすさを整える

この順にすると、構造の根本原因を先に潰しやすくなります。
