このサイトは、**Re:VIEW の公式仕様**、**著者 repo の実例**、**Nextpublishing 固有の diagnostics** を、初心者が順番に追えるように並べ直した docs hub です。

Re:VIEW に慣れていない段階では、検索語が分からないまま詰まることがよくあります。たとえば「見出しが悪い」のか「ID が重複した」のか「表の書き方が違う」のかが切り分けられないと、公式 docs を開いてもどこから読めばよいか決めにくくなります。そこで本サイトでは、**先に全体像を押さえ、そのあと細かい syntax と policy へ進む** 読み順を採っています。

## この docs hub の責務

### 公式 Re:VIEW と派生 docs hub の境界

公式 Re:VIEW docs は、構文と設定の一次根拠です。このサイトはその内容を置き換えるものではなく、**初心者が迷わず読むための導線** を加えたものです。

この違いを最初に押さえておくと、「この説明は公式仕様なのか、それとも現場での運用ルールなのか」を取り違えにくくなります。

- 構文の正しさを決めるのは公式 Re:VIEW docs
- よくある author repo の形を説明するのは実例観測
- Nextpublishing 固有の差し戻し条件を説明するのは diagnostics corpus

### なぜ 3 種類の根拠を分けるのか

初心者向けの説明では、つい「普通はこう書く」とまとめてしまいがちです。しかし実際には、次の 3 種類は役割が違います。

| 根拠 | 役割 | 典型的な問い |
| --- | --- | --- |
| Normative | 構文や設定の正否を決める | この書き方は Re:VIEW として正しいか |
| Descriptive | 実務でよく見る形を説明する | 著者 repo はだいたいどう分かれているか |
| Policy | 組版や運用上の制約を決める | Nextpublishing では何が差し戻しになるか |

この 3 つを混ぜてしまうと、「公式では通るが Nextpublishing では注意が必要」という場面が説明しづらくなります。

### このサイトと書籍派生物の使い分け

公開サイトは、**いま必要な章へ飛ぶ** ための形です。書籍派生物は、**順番に通読する** ための形です。内容の元は同じでも、読ませ方は同じにしません。

> 迷ったら、まず公開サイトの章から読む。まとまって学び直すときに書籍派生物を見る。  
> これがこの repo の基本運用です。

## 読者別の入口

### 著者が最短で修正したいとき

差し戻しや preview error をすぐ直したいときは、次の順で読むと効率が良いです。

1. 第7章「トラブルシューティング」
2. 第6章「Nextpublishing 固有ルールと diagnostics」
3. 第3章「Re:VIEW 原稿の基本記法」

この順にすると、まず症状から入り、次に rule を確認し、最後に構文の基本へ戻れます。

### Re:VIEW を体系的に学びたいとき

ゼロから読むなら、次の順を勧めます。

1. 第2章「著者リポの全体像」
2. 第3章「Re:VIEW 原稿の基本記法」
3. 第4章「図表・脚注・コード・数式」
4. 第5章「config.yml と catalog.yml の基本」

先に repo の骨格を見ておくと、構文が「どこに置かれるものか」が分かるようになります。

### サポート担当が根拠を逆引きしたいとき

問い合わせ対応では、いきなり説明を書くより、先に根拠をそろえたほうが安全です。

- 現象を第7章で探す
- 診断コードや禁止事項を第6章で確認する
- 仕様上の根拠を第8章の source trail から辿る

## 先に知っておく前提

### Re:VIEW author repo を前提にしている

このサイトは単一の `.re` ファイルだけで完結する書き方ではなく、**`config.yml`、`catalog.yml`、複数の章ファイル、`images/`** を持つ author repo を前提にしています。

### diagnostics-first で考える

初学者ほど、「自分の書き方が悪いのか」「運用上の制約なのか」で迷います。そこで本サイトでは、**現象 -> diagnostics -> rule -> syntax** の順で確認できるようにしています。

### すべてを一度に覚えなくてよい

Re:VIEW は書けることが多い反面、最初から全部を覚える必要はありません。まずは次の 4 つが分かれば、執筆はかなり進められます。

- 見出し
- 箇条書き
- コード例
- 図表と脚注

## この章の終わりに

次の章では、author repo 全体を見渡して、`catalog.yml` が何を束ね、章ファイルがどう並ぶかを整理します。ここが見えると、その後の syntax が「部品」として理解しやすくなります。

## この章の根拠

- Normative: Re:VIEW quickstart
- Descriptive: sample author repo と production-like snapshot の章構成観測
- Policy: diagnostics-first 導線の必要性は Nextpublishing rule 説明との接続要件
