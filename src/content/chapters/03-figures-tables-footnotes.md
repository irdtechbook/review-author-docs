図表系は、本文よりも「周辺条件」で壊れやすい領域です。画像ファイル名、説明文、ID、表セル区切りなど、本文とは別の注意点が増えます。ここでは、初心者が最初に外しやすいポイントから整理します。

## 画像の扱い

### 画像は本文と別管理で考える

画像は `.re` の中に埋め込まれているのではなく、別ファイルとして置かれます。だから、原稿だけ見ていると「なぜ図が出ないのか」に気づきにくくなります。

画像で見るべき点は次の 3 つです。

- 画像ファイルが正しい場所にあるか
- 本文側の参照 ID が合っているか
- 運用上の拡張子制約に引っかかっていないか

### `//image` と `@<img>` の役割を分ける

初心者は「画像を置く命令」と「画像を参照する命令」を混同しがちです。まずは、**画像のまとまり記法と画像参照は別物** と覚えてください。

### 画像の確認は本文だけで終わらせない

画像不具合は、本文が正しくてもファイル名の違いで起きます。本文修正だけで直そうとせず、画像ファイル側まで見る習慣を付けてください。

## 表の扱い

### 表は見た目より繊細

表は、書いている最中はそれらしく見えても、区切り記号の違いで壊れます。とくに Nextpublishing の診断では、セル区切りの扱いが厳しめです。

### まず覚えるべき表の確認点

| 確認点 | なぜ重要か | ありがちな事故 |
| --- | --- | --- |
| セル区切り | ビルド時の表構造に直結する | スペース区切りで書いてしまう |
| 説明文 | 読者と参照の両方に効く | 省略して後で意味が分からなくなる |
| ID | 参照の安定性に効く | 重複や空 ID に気づかない |

### 表は短く保つ

はじめて書籍化する著者の原稿では、表に情報を詰め込みすぎないことが大切です。長い説明は本文に戻し、表は「比較軸をひと目で見せる」用途に絞ると読みやすくなります。

## 脚注・注記・補足

### 脚注は本文を軽くするために使う

脚注は便利ですが、何でも脚注に送ると本文が弱くなります。先に本文で理解できるように書き、補足だけを脚注へ回すのが基本です。

### 参照のつながりを意識する

脚注のまとまり記法と脚注参照がずれると、読者にもビルドにも優しくありません。脚注を追加したら、本文側の参照まで見直してください。

### 注記の書きすぎに注意する

注記、脚注、コラムを多用すると、初心者には「どこが本筋か」が見えにくくなります。重要度が高い説明は本文で扱い、補足だけを外に出すのが基本です。

## コード・数式・図式

### コード例は短く切る

はじめて書籍化する著者には、長いコードを一度に見せるより、**例を短くして説明を近くに置く** ほうが理解されます。

```text
//cmd{
$ review-compile chapter.re
//}
```

### 数式は `//texequation` でまとめる

数式を入れたいときは、まず `//texequation` を使う書き方を覚えると整理しやすくなります。

```text
//texequation[eq01][オイラーの公式]{
e^{i\theta}=\cos\theta+i\sin\theta
//}
```

このサイトでは、まず **TeX 表記のまま読める形で紹介** し、VS Code の WebView preview では数式 preview も確認できる、という順で理解すると迷いにくくなります。

### 数式は「必要な箇所だけ」に絞る

数式は便利ですが、本文のどこにでも入れるものではありません。式を入れたほうが説明が短く、正確になる場所だけで使うと、初学者にも読みやすい原稿になります。

## Mermaid の使いどころ

### Mermaid は流れを短く見せたいときに向く

Mermaid は、手順や関係を短く見せたいときに向いています。複雑な構成全体を全部図にするより、「何が何に依存するか」を 1 つずつ見せるほうが初学者には親切です。

```text
//graph[build-flow][mermaid][確認の流れ]{
flowchart TD
  A[原稿を書く] --> B[HTML ビルド]
  B --> C[WebView で確認]
  C --> D[修正する]
//}
```

### Mermaid の例

<div class="mermaid">
flowchart TD
  A[原稿を書く] --> B[HTML ビルド]
  B --> C[WebView で確認]
  C --> D[修正する]
</div>

### Mermaid は小さく保つ

`dev/vscode` 側の preview と診断辞書でも、Mermaid は複雑になるほど warning が出やすい前提で扱われています。はじめての原稿では、縦に長すぎる図や要素を詰め込みすぎた図を避け、1 図 1 目的で使うほうが安全です。
