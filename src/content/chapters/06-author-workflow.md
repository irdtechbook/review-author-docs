個別のルールを知っていても、作業順が悪いと同じ場所で何度もつまずきます。この章では、**執筆 -> ビルド / プレビュー -> 診断 -> 修正** を 1 本の流れとして整理します。

## 著者ワークフロー

### 章追加前に決めること

章を書き始める前に、最低限これだけは決めてください。

1. その章を `catalog.yml` のどこへ置くか
2. 画像や表が多い章かどうか
3. 既存章との参照関係があるか

この 3 点を決めてから書き始めると、あとで構造修正が減ります。

### 原稿執筆中に維持する粒度

初心者は、一度に長い区間を書き切りたくなります。しかしビルドやプレビューを考えると、**中見出し 1 つぶん書いたら確認する** くらいの粒度が扱いやすいです。

### 図表と参照を後で壊さない書き方

図表や参照は、本文が進んでから一気に足すと崩れやすくなります。次のどちらかで進めると安全です。

- 先に ID だけ仮置きしておく
- 図表が必要な節を短く区切り、節ごとに確認する

## 手元での確認方針

### まず TeX 組版を目標にしない

TeX 環境の構築は重く、はじめての著者が最初に抱える作業としては負担が大きすぎます。Nextpublishing 組版を前提にしていても、**著者の手元で最初に必要なのは HTML ビルドが通ること** です。

### VS Code の WebView preview を使う

`dev/vscode` の Re:VIEW preview 拡張は、`config.yml` / `catalog.yml` を見ながら、WebView で紙面に近い確認ができる前提で作られています。執筆中は、まずこの preview で見出し、図表、脚注、Mermaid、数式 preview を確認する流れで十分です。

### HTML と WebView で確認する順番

<div class="mermaid">
flowchart LR
  A[原稿を書く] --> B[HTML ビルドを通す]
  B --> C[VS Code WebView で確認する]

最終的な組版結果そのものは Nextpublishing 側で確認するとしても、著者側の作業はこの順番でかなり前へ進められます。

## 診断から始める作業の流れ

### まず診断コードからルールを引く

現象からルールを想像すると、外すことがあります。診断が出ているなら、先にその ID からルールを引き、何が禁じられているかを確認します。

### 根本原因と見えている症状を分ける

たとえば「表が崩れる」という症状があっても、原因は説明文ではなくセル区切りかもしれません。はじめて書籍化する著者は、**見えている現象と修正すべき根を分ける** ことが重要です。

### 修正後に再確認する観点

1 回警告が消えたら終わり、ではありません。次の 3 点を見直してください。

1. 参照が新たに壊れていないか
2. 見出しや番号付けが不自然になっていないか
3. 読みやすさを損なっていないか

### Mermaid と数式は preview 前提で見る

`dev/vscode` 側では Mermaid は WebView 上で SVG として描画され、`//texequation` は KaTeX で preview されます。ただし、Mermaid は環境差で見え方が変わることがあるので、複雑な図は作り込みすぎず、数式も短い単位で確認するほうが安全です。

## 自分で切り分けるときの流れ

### 症状をどの種類か決める

修正を始める前に、まず困りごとを次のどれかに当てはめると進めやすくなります。

- 構造の問題
- 記法の問題
- 運用ルールや診断の問題
- 周辺ファイルの問題

### 参考資料を見ながら整理する

迷ったときは、次の順で整理すると判断しやすくなります。

- 公式仕様ではどうか
- 運用上はどうか
- だから今はどう直すか

### このガイドに知見を足すべき条件

毎回このガイドを更新する必要はありません。ただし、次のようなものは反映価値があります。

- 同じ質問が繰り返される
- 診断の解き方が初心者には分かりづらい
- 既存章の読み順では辿りにくい

この流れを覚えておくと、詰まったときに「次に何を見るか」が決まります。

## はじめて書籍化する著者向けの作業メモ

- 大きく書いて、あとで直すより、小さく書いて頻繁に確認する
- 診断は失敗通知ではなく、修正の入口と考える
- 1 回で完全に直そうとせず、原因の層を 1 つずつ減らす
