個別の rule を知っていても、作業順が悪いと同じ場所で何度もつまずきます。この章では、**執筆 -> build / preview -> diagnostics -> 修正** を 1 本の流れとして整理します。

## 著者ワークフロー

### 章追加前に決めること

章を書き始める前に、最低限これだけは決めてください。

1. その章を `catalog.yml` のどこへ置くか
2. 画像や表が多い章かどうか
3. 既存章との参照関係があるか

この 3 点を決めてから書き始めると、あとで構造修正が減ります。

### 原稿執筆中に維持する粒度

初心者は、一度に長い区間を書き切りたくなります。しかし build や preview を考えると、**中見出し 1 つぶん書いたら確認する** くらいの粒度が扱いやすいです。

### 図表と参照を後で壊さない書き方

図表や参照は、本文が進んでから一気に足すと崩れやすくなります。次のどちらかで進めると安全です。

- 先に ID だけ仮置きしておく
- 図表が必要な節を短く区切り、節ごとに確認する

## diagnostics-first workflow

### まず診断コードからルールを引く

現象から rule を想像すると、外すことがあります。diagnostics が出ているなら、先にその ID から rule を引き、何が禁じられているかを確認します。

### root cause と symptom を分ける

たとえば「表が崩れる」という symptom があっても、原因は caption ではなくセル区切りかもしれません。初心者向けには、**見えている現象と修正すべき根を分ける** ことが重要です。

### 修正後に再確認する観点

1 回 warning が消えたら終わり、ではありません。次の 3 点を見直してください。

1. 参照が新たに壊れていないか
2. 見出しや番号付けが不自然になっていないか
3. 読みやすさを損なっていないか

## support / maintainer workflow

### 問い合わせから該当ページへ誘導する

サポートでは、いきなり回答を書き始めるより、まず問い合わせを次のどれに当てはめるかを決めると楽になります。

- 構造の問題
- syntax の問題
- policy / diagnostics の問題
- 周辺ファイルの問題

### 根拠 source を明示して説明する

答えを返すときは、可能なら次の形で示すと再利用しやすくなります。

- 公式仕様ではどうか
- 運用上はどうか
- だから今はどう直すか

### docs hub 側へ知見を反映する条件

問い合わせのたびに docs を更新する必要はありません。ただし、次のようなものは反映価値があります。

- 同じ質問が繰り返される
- diagnostics の解き方が初心者には分かりづらい
- 既存章の読み順では辿りにくい

## 1 本の流れで見る

この流れを覚えておくと、詰まったときに「次に何を見るか」が決まります。

## 初心者向けの作業メモ

- 大きく書いて、あとで直すより、小さく書いて頻繁に確認する
- diagnostics は失敗通知ではなく、修正の入口と考える
- 1 回で完全に直そうとせず、原因の層を 1 つずつ減らす

## この章の終わりに

次章では、実際の症状から逆引きできる troubleshooting に入ります。検索語が分からない状態でも辿れるように、症状の見え方から整理します。

## この章の根拠

- Normative: build / preview の基本的な位置づけ
- Descriptive: author repo 運用での章追加と確認の流れ
- Policy: diagnostics-first の必要性
