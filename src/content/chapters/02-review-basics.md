ここでは、最初に触る頻度が高い記法だけを先にまとめます。すべての tag を暗記するより、**壊れやすい基本を外さない** ことを優先します。

## 段落と見出しの基本

### 段落は空行で区切る

Markdown に慣れている人ほど、改行だけで段落を分けたくなります。しかし Re:VIEW では、**段落の切れ目は空行で明示する** と考えるほうが安全です。

```text
= 章タイトル

これは最初の段落です。

これは次の段落です。
```

### 見出しは `=` から始まる

見出しレベルは `=` の数で表します。まずは次の理解で十分です。

- `=`: 章
- `==`: 節
- `===`: 小節

深いレベルまで使えるとしても、初心者向けの原稿ではむやみに深くしないほうが読みやすくなります。

### 見出し行に余計な装飾を入れない

見出しは目立たせたくなりますが、見出しの中に複雑な inline tag を入れると、policy 側で嫌われることがあります。まずは plain text で書き、必要な強調は本文側に逃がすのが安全です。

## inline 記法の基本

### inline は「行の中で効く部品」

inline は、文の流れを切らずに使う部品です。代表例は次のとおりです。

- `@<code>{...}`: コードや識別子を文中で示す
- `@<fn>{...}`: 脚注参照
- `@<img>{...}`: 画像参照

```text
設定ファイルの名前は @<code>{config.yml} です。
```

### 何でも inline で済ませない

初心者がやりがちなのは、長いコードや強調を全部 inline に押し込むことです。行の中に収まらないものは block に分けたほうが読みやすく、policy 側でも扱いやすくなります。

### 「公式で書ける」と「運用で勧める」は別

ここが混線しやすいところです。公式に存在する記法でも、Nextpublishing 運用では避けたいものがあります。その切り分けは第6章で整理します。

## block 記法の基本

### block はかたまりで置く

コード例、引用、表、注記などは block として置きます。初心者はまず、**1 行で済ませるか、まとまりとして置くか** を意識すると整理しやすくなります。

### コード例は読み手の目的で分ける

同じコードでも、目的によって block を使い分けます。

- 行番号付きで説明したい: `//listnum`
- ただ見せたい: `//emlist`
- コマンドとして見せたい: `//cmd`

### block を増やしすぎない

block は便利ですが、多用すると原稿が断片化します。初心者向けには、「本文で説明 -> 短い例を 1 つ」くらいのテンポが読みやすいです。

## 箇条書きとネストの注意点

### 箇条書きは空白を丁寧に扱う

箇条書きは見た目が単純なので油断しやすいですが、空白や空行の扱いで diagnostics が出やすい領域です。

書くときは次の 3 点を意識します。

1. 行頭の空白を崩さない
2. 本文から箇条書きへ入る前に必要な空行を入れる
3. ネストを深くしすぎない

### ネストは説明の逃げ道ではない

論点が増えたときに、箇条書きを深くネストして整理したくなることがあります。しかし深くなるほど読者は見失います。初心者向けの説明では、ネストで頑張るより**見出しを切る** ほうが理解しやすいです。

### 最低限の書き方

```text
== 確認ポイント

 * config.yml がある
 * catalog.yml がある
 * 章ファイルが catalog.yml に入っている
```
